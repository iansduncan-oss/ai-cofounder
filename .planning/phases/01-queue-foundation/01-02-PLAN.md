**Step 1 — Create `apps/agent-server/src/worker.ts`:**

This is a standalone Node.js entry point (not Fastify). It bootstraps the same services as `server.ts` but without HTTP. Key pattern: mirror the service construction from `server.ts` / `buildServer()`.

```typescript
// apps/agent-server/src/worker.ts
// Standalone worker process — same Docker image, different CMD
// Processes agent-task jobs from the BullMQ queue

import { createLogger, requireEnv, optionalEnv } from "@ai-cofounder/shared";
import { createDb, runMigrations } from "@ai-cofounder/db";
import {
  getRedisConnection,
  startWorkers,
  stopWorkers,
  closeAllQueues,
} from "@ai-cofounder/queue";
import { createLlmRegistry } from "./server.js";
import { TaskDispatcher } from "./agents/dispatcher.js";
import { VerificationService } from "./services/verification.js";
import { createEmbeddingService } from "@ai-cofounder/llm";
import { createSandboxService } from "@ai-cofounder/sandbox";
import { createWorkspaceService } from "./services/workspace.js";
import { createNotificationService } from "./services/notifications.js";

const logger = createLogger("worker");

async function main() {
  logger.info("Worker starting...");

  // Initialize Redis connection
  const redisUrl = requireEnv("REDIS_URL");
  getRedisConnection(redisUrl);

  // Initialize database (with migrations)
  const databaseUrl = requireEnv("DATABASE_URL");
  const db = createDb(databaseUrl);
  await runMigrations(db);

  // Bootstrap services (same as server.ts but no Fastify)
  const llmRegistry = createLlmRegistry();
  const geminiKey = optionalEnv("GEMINI_API_KEY", "");
  const embeddingService = geminiKey ? createEmbeddingService(geminiKey) : undefined;
  const sandboxService = createSandboxService();
  const workspaceService = createWorkspaceService();
  const notificationService = createNotificationService();

  const verificationService = new VerificationService(
    llmRegistry,
    db,
    notificationService,
    workspaceService,
    sandboxService,
  );

  const dispatcher = new TaskDispatcher(
    llmRegistry,
    db,
    embeddingService,
    sandboxService,
    notificationService,
    workspaceService,
    verificationService,
  );

  // Start agent-task worker ONLY (monitoring/briefing/notification stay in HTTP server)
  startWorkers({
    agentTask: async (job) => {
      const { goalId, userId } = job.data;
      logger.info({ jobId: job.id, goalId, userId }, "Processing agent task");

      try {
        await dispatcher.runGoal(goalId, userId);
        logger.info({ jobId: job.id, goalId }, "Agent task completed");
      } catch (err) {
        logger.error({ jobId: job.id, goalId, err }, "Agent task failed");
        throw err; // Re-throw so BullMQ marks job as failed and handles retries
      }
    },
  });

  logger.info("Worker started — waiting for jobs");

  // Graceful shutdown handler
  const shutdown = async (signal: string) => {
    logger.info({ signal }, "Shutdown signal received — draining active jobs...");
    await stopWorkers();       // waits for active job to finish
    await closeAllQueues();    // close queue connections
    logger.info("Worker shut down cleanly");
    process.exit(0);
  };

  process.on("SIGTERM", () => shutdown("SIGTERM"));
  process.on("SIGINT", () => shutdown("SIGINT"));
}

main().catch((err) => {
  logger.error({ err }, "Worker startup failed");
  process.exit(1);
});
```

Important notes:
- `createLlmRegistry()` is imported from `./server.js` (it's already exported there).
- The worker only registers `agentTask` processor — NOT monitoring/notification/briefing/pipeline.
- `requireEnv("REDIS_URL")` will throw if REDIS_URL is missing (worker cannot operate without Redis).
- `requireEnv("DATABASE_URL")` will throw if DATABASE_URL is missing.
- The worker catches errors in the processor and re-throws them so BullMQ can handle retries.
- The Dockerfile already builds worker.ts since it compiles the entire `apps/agent-server/` workspace.

**Step 1 — Update `apps/agent-server/src/routes/execution.ts`:**

Replace the blocking `POST /:id/execute` handler with a non-blocking version that enqueues to BullMQ and returns 202.

The updated handler should:
1. Import `enqueueAgentTask` from `@ai-cofounder/queue` and `getGoal`, `updateGoal` (or use raw Drizzle update) from `@ai-cofounder/db`.
2. Look up the goal to get its description/title and validate it exists.
3. Call `enqueueAgentTask({ goalId: id, prompt: goal.description ?? goal.title, userId, priority: request.body.priority })`.
4. Store the returned `jobId` in `goal.metadata.queueJobId` using a Drizzle update: spread existing metadata, add `queueJobId`.
5. Return `reply.status(202).send({ jobId, status: "queued", goalId: id })`.

Add a `priority` field to the request body type: `Body: { userId?: string; webhookUrl?: string; priority?: "critical" | "high" | "normal" | "low" }`.

For the `webhookUrl` parameter: store it in `goal.metadata.webhookUrl` alongside `queueJobId`. The worker can read it later when Phase 2 adds event publishing. For now, just store it.

**Keep the SSE streaming endpoint (`GET /:id/execute/stream`) as-is.** Per STATE.md: "SSE streaming is the highest-risk integration — plan Phase 2 carefully." The SSE endpoint will still work for in-process execution during the transition but won't receive events from the worker. This is acceptable for Phase 1.

**Keep the `GET /:id/progress` endpoint as-is** — it reads from dispatcher's in-memory state, which will be populated when the worker runs.

Import the `goals` table and `eq` from Drizzle for the metadata update:
