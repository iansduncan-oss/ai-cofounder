  - `onClose` hook: call `subscriber.quit()` and `redisPubSub.close()`.

**Step 3: Rewrite SSE endpoint in `apps/agent-server/src/routes/execution.ts`.**

CRITICAL COMPATIBILITY NOTE: The dashboard's `useSSE` hook uses `source.onmessage` (the default event handler). Named SSE events (with `event: xyz` field) do NOT trigger `onmessage` — they require `addEventListener('xyz', ...)`. The current endpoint uses named events which is technically incompatible, but we must ensure the new endpoint works correctly with the dashboard.

Solution: Write SSE data WITHOUT the `event:` field prefix. Just write `data: ${JSON.stringify(payload)}\n\n`. This triggers the default `message` event type which `onmessage` receives. The dashboard reads `data.status` to detect completion/failure.

Replace the `GET /:id/execute/stream` handler body entirely. Remove the inline `dispatcher.runGoal()` call. The new handler:

1. Set SSE headers: `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`.

2. Define `send(data)` helper: writes `data: ${JSON.stringify(data)}\n\n` to `reply.raw`. Check `!reply.raw.writableEnded` before writing. NO `event:` field — critical for useSSE compatibility.

3. **Replay history** — call `app.redisPubSub.getHistory(goalId)`. For each event:
   - If it's a lifecycle event with `type === "job_completed"`: send it with `status: "completed"` and `reply.raw.end()`. Return early (job already finished).
   - If it's a lifecycle event with `type === "job_failed"`: send it with `status: "failed"` and `reply.raw.end()`. Return early.
   - If it's a progress event (has `taskId`): send it as-is (it already has `status: "started"|"completed"|"failed"` from the dispatcher).

4. **Subscribe to live events** — call `app.subscribeGoal(goalId)`. Set up a listener on `app.agentEvents.on(channel, onMessage)` where `channel = goalChannel(goalId)`.

   The `onMessage` handler parses JSON, then:
   - If lifecycle `job_completed`: send with `status: "completed"`, call `cleanup()`.
   - If lifecycle `job_failed`: send with `status: "failed"`, call `cleanup()`.
   - If progress event: send as-is.

5. **Cleanup function** — removes EventEmitter listener via `app.agentEvents.off(channel, onMessage)`, calls `app.unsubscribeGoal(goalId)` (non-fatal catch), ends response if not already ended.

6. **Client disconnect** — `reply.raw.on("close", cleanup)`. This is CRITICAL for preventing EventEmitter listener leaks. The existing endpoint uses `try/finally` which does NOT detect client disconnects.

Remove the `dispatcher` and `verificationService` instantiation from the route plugin IF they are no longer used by any route in this file. The `POST /:id/execute` route (BullMQ enqueue) and `GET /:id/progress` route do NOT use the dispatcher anymore since Phase 1 moved execution to the worker. Check carefully: `GET /:id/progress` calls `dispatcher.getProgress(id)` — if so, keep the dispatcher instantiation but remove the `VerificationService` import if unused.

Actually, on inspection, `GET /:id/progress` does call `dispatcher.getProgress(id)`. Keep the dispatcher. Only remove imports/code that are truly unused after the SSE rewrite.
