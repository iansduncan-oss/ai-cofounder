Two endpoints:

2. `PUT /:toolName` — accepts body `{ tier: "green"|"yellow"|"red", timeoutMs?: number }` with TypeBox validation. After DB write, calls `app.autonomyTierService.reload()` to refresh in-memory cache. Returns the updated record. This is how AUTO-05 "take effect immediately" works.

Follow the existing route pattern: `export async function autonomyRoutes(app: FastifyInstance) { ... }`. Use TypeBox `Type.Object()` for request/response schemas. Tag: "autonomy".

**Add approval timeout sweep in `apps/agent-server/src/plugins/queue.ts`:**
Add a recurring BullMQ job (every 60 seconds) named `approval-timeout-sweep`. The worker:
1. Calls `listExpiredPendingApprovals(db)` — returns approvals where status is 'pending' and createdAt is older than a threshold.
2. For each expired approval, calls `resolveApproval(db, approval.id, "rejected", "Auto-denied: approval timeout exceeded")`.
3. Logs count of expired approvals.

This handles the case where the server restarts while a yellow-tier tool is waiting — the sweep catches orphaned approvals.

NOTE: The `listExpiredPendingApprovals` function from Plan 01 uses a fixed 300s default. For more accurate per-tool timeouts, the sweep job should join `approvals` with `toolTierConfig` to get the per-tool `timeoutMs`. However, since the approval record does not currently store the tool name, use the fixed default for now. This is acceptable because the in-process polling in `executeWithTierCheck` already uses the correct per-tool timeout — the sweep job is a safety net for orphaned approvals after restarts.
