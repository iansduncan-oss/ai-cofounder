## 2. Key Technical Decisions & Outcomes

### Architecture Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| **Fire-and-forget for RAG ingestion** | `.catch(() => {})` on all enqueue calls prevents blocking agent responses | Good — zero latency impact on user-facing responses |
| **In-memory Map for autonomy tiers** | DB-backed with in-memory cache, `reload()` on dashboard changes | Good — instant enforcement, no DB round-trip per tool call |
| **onCompletion hook on LlmRegistry** | Automatic cost recording on every LLM call vs manual `recordLlmUsage()` | Excellent — eliminated 5 manual call sites, 100% coverage |
| **DB-first config reads** | `getAppSetting(db, key) ?? parseFloat(optionalEnv(name, '0'))` | Good — settings configurable at runtime without restart |
| **useSyncExternalStore for localStorage** | React 18 pattern for external store sync | Good — no stale reads vs useState+useEffect |
| **Positional params for Orchestrator** | 11+ constructor params passed positionally | Pain point — fragile, hard to extend. Should be an options object. |
| **Dynamic import() for AutonomousExecutorService** | Avoids circular dependency in autonomous-session.ts | Acceptable hack — works but obscures dependency graph |
| **mockDbModule() pattern** | Replaced Proxy-based mocks that hung in vitest 4.x | Critical fix — unblocked all test files |

### Infrastructure Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| **Tailscale v4 OAuth for CI deploy** | GitHub deprecated authkey-based action | Painful migration — ACL policy issues took 2 sessions to resolve |
| **BullMQ for all background work** | Unified queue system for monitoring, briefings, budgets, autonomous sessions | Good — consistent patterns, Redis-backed, priority support |
| **Redis distributed lock for scheduling** | Only one autonomous session at a time | Good — simple, reliable, self-healing via TTL |
| **Drizzle migrations tracked in DB** | `__drizzle_migrations` table for prod sync | Good after VPS seeding fix — was blocking deploys before session 20 |

## 3. What Went Well

### Velocity
- 10 phases in 7 days is fast. The GSD framework (research → plan → execute → verify) kept momentum high.
- Plans averaged 3–15 minutes execution time. Short feedback loops.
- Parallel phase execution (8+9, 11+12+13, 14+15) cut calendar time significantly.

### Test Coverage
- Grew from ~1100 to ~1289+ tests across 96+ test files.
- `mockDbModule()` pattern from `@ai-cofounder/test-utils` made test authoring consistent and fast.
- Test failures caught real bugs (journal useMemo side effect, mock isolation issues).

### Requirement Traceability
- Every requirement mapped to a phase, every phase to plans, every plan to a summary.
- Milestone audit (Phase 17) caught 7 integration gaps that would have shipped broken. The audit-then-fix pattern works.

### Architectural Patterns
- `onCompletion` hook for cost tracking was the cleanest design in the milestone. Zero manual instrumentation.
- Three-tier autonomy system (green/yellow/red) is simple to understand and extend.
- Project registry with per-project WorkspaceService instances is clean isolation.

### Documentation
- 43 requirements fully specified before any code was written.
- Dependency graph drove execution order — no blocked phases, no rework.

## 4. What Was Painful

### Orchestrator Constructor Bloat
The Orchestrator constructor grew to 11+ positional parameters across phases 8–17. Every new service (autonomyTierService, projectRegistryService, monitoringService, messagingService) added another positional arg. This made:
- Wiring fragile — wrong position = silent bugs
- Test setup verbose — every mock must be in the right slot
- Phase 17 specifically hit this: plan said "param 12" but reality was "param 9"

**Lesson:** Refactor to an options/config object before v3.0.

### Vitest Mock Isolation
- Proxy-based `vi.mock("@ai-cofounder/db")` caused ESM resolution hangs in vitest 4.x. Took a full session (9a) to diagnose and fix across all 40+ test files.
- `mockResolvedValueOnce` vs `mockResolvedValue` caused non-obvious failures when BullMQ seed calls consumed the mock value before the test's route call.
- `vi.advanceTimersByTimeAsync` vs `runAllTimersAsync` — the latter caused infinite loops with setInterval-based schedulers.

**Lesson:** Document mock patterns in CLAUDE.md (done). Consider a test harness that enforces the pattern.

### Deploy Pipeline Fragility
- Tailscale v3 → v4 migration broke deploys for 2 sessions.
- `GITHUB_SHA` wasn't expanding inside single-quoted SSH heredoc — subtle shell quoting bug.
- VPS drizzle tracking table needed manual seeding of 9 missing entries.
- Each deploy issue required a full CI cycle to verify (slow feedback).

**Lesson:** Deploy pipeline needs local smoke-test capability. Consider `act` for local GitHub Actions testing.

### Phase 11/12 Documentation Gaps
- Phase 11 (Autonomous Scheduling) has plans and validation but no SUMMARY files.
- Phase 12 (Work Journal) has no phase directory at all — built directly without GSD tracking.
- Both phases work correctly, but the documentation trail is incomplete.

**Lesson:** Don't skip summaries even for "quick" phases. The retrospective suffers.

### Dashboard State Management
- `useSyncExternalStore` for localStorage was the right call but non-obvious.
- Multiple hooks needed careful coordination (useActiveProject, useSettings, useProjects).
- Date-range filtering done client-side works for now but won't scale.

## 7. Lessons for v3.0 Planning

### Do Again
1. **Requirements-first planning** — 43 requirements written before any code. Every phase had clear success criteria. Zero scope ambiguity.
2. **Dependency graph driving execution order** — parallel tracks (8+9, 11+12+13) saved days.
3. **Milestone audit before declaring done** — Phase 17 caught 7 real gaps. Make this a standard step.
4. **mockDbModule pattern** — shared test infrastructure pays dividends. Invest in test-utils early.
5. **onCompletion hooks** — cross-cutting concerns (cost tracking, logging) should be hooks, not manual calls.

### Do Differently
1. **Refactor Orchestrator to options object** — 11 positional params is unmaintainable. Do this before v3.0 adds more services.
2. **Write SUMMARY files for every plan** — even "quick" phases. The documentation gap for Phase 11/12 made this retrospective harder.
3. **Local deploy testing** — the Tailscale/SSH/heredoc bugs burned 2 sessions. Need `act` or equivalent.
4. **Server-side pagination for journal/notifications** — client-side filtering won't scale past ~1000 entries.
5. **Consider integration tests** — unit test coverage is strong but E2E flow testing (the gaps Phase 17 closed) needs automated coverage.
6. **Budget for tech debt** — lint cleanup, mock pattern migration, deploy fixes all happened reactively. Allocate a "stabilization" phase in v3.0.

### Watch Out For
- **Constructor/DI complexity** — more services = more wiring pain. Consider a proper DI container or builder pattern.
- **Dashboard state proliferation** — 7 new pages means 7 more query hooks, mutation hooks, and state sync points. Consider a state machine or reducer pattern.
- **Queue job proliferation** — monitoring, briefings, budgets, autonomous sessions, DLQ checks all on BullMQ. Need queue health monitoring.
- **LLM cost scaling** — autonomous execution + scheduled sessions + briefings + consolidation = significant token burn. Budget enforcement is built (FIN-03) but needs real-world tuning.

## 8. What v2.0 Means

The AI Cofounder went from a reactive chat agent to an autonomous engineering partner:

- **It remembers** — conversations, decisions, and project context persist across sessions (MEM, SESS)
- **It governs itself** — three-tier approval system controls what it can do without asking (AUTO)
- **It works independently** — picks up tasks, writes code, runs tests, commits, opens PRs (TERM)
- **It works on schedule** — recurring execution loop with safety guardrails (SCHED)
- **It tracks its own work** — browsable journal, daily standups, cost attribution (JRNL, FIN)
- **It manages multiple projects** — project registry with per-project context and VPS awareness (PROJ)
- **It automates content** — pipeline templates, n8n integration, scheduled content creation (CONT)
- **It has a command center** — dashboard surfaces everything in one place (DASH)

43 requirements. 10 phases. 24 plans. 7 days. Zero requirements dropped.
