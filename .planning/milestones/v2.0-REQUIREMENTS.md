# Requirements: v2.0 Autonomous Cofounder

**Defined:** 2026-03-09
**Core Value:** Transform the AI Cofounder from a reactive tool into an autonomous engineering partner that works independently, remembers everything, and manages all systems.

**Total:** 43 requirements across 10 categories

---

## Autonomous Execution (TERM)

Agent works through planned tasks end-to-end without human trigger.

- [x] **TERM-01**: Agent picks up tasks from the goal backlog and executes them without requiring a human trigger
- [x] **TERM-02**: Agent chains workspace, git, and sandbox tools to complete coding tasks end-to-end (read code → edit → test → commit)
- [x] **TERM-03**: Agent commits changes with conventional commit messages that link back to the originating goal/task
- [x] **TERM-04**: Agent creates PRs for completed work with auto-generated descriptions summarizing changes
- [x] **TERM-05**: Each execution produces a structured work log entry (task, actions taken, outcome, duration, token cost)

## Memory & Context (MEM)

Persistent memory that makes every conversation build on every previous one.

- [x] **MEM-01**: All conversations auto-ingested into RAG vector store after completion (chunked, embedded, indexed)
- [x] **MEM-02**: Decisions stored with tagged context — what was decided, why, and what alternatives were considered
- [x] **MEM-03**: Project context (architecture docs, conventions, patterns) ingested from repo documentation on registration
- [x] **MEM-04**: Agent auto-loads relevant memories before starting any task (no manual "recall" required)
- [x] **MEM-05**: Periodic memory consolidation — related memories summarized into coherent knowledge entries

## Autonomy Tiers (AUTO)

Three-tier approval system controlling what the agent can do without asking.

- [x] **AUTO-01**: Green tier — agent executes without approval (read-only ops, research, memory, code generation, sandbox execution)
- [x] **AUTO-02**: Yellow tier — agent requests approval before executing (git push, deploy, DB mutations, external API calls, PRs)
- [x] **AUTO-03**: Red tier — agent never executes without explicit human authorization (delete data, production DB changes, financial operations)
- [x] **AUTO-04**: Approval requests delivered via Slack/Discord with full context and one-tap approve/deny buttons
- [x] **AUTO-05**: Tier assignments configurable per-tool from a dashboard settings page

## Work Journal & Standup (JRNL)

Browsable activity log with proactive daily check-ins.

- [x] **JRNL-01**: Agent produces daily standup summary — what was done yesterday, what's planned today, blockers
- [x] **JRNL-02**: All agent actions logged in a browsable work journal with timestamps, tool calls, and outcomes
- [x] **JRNL-03**: Work journal entries grouped by goal/task with links to relevant PRs, commits, and artifacts
- [x] **JRNL-04**: Work journal accessible from dashboard with date-range filtering and full-text search

## Content Automations (CONT)

YouTube pipeline and n8n workflows managed as first-class agent tasks.

- [x] **CONT-01**: YouTube Shorts pipeline registered as a managed pipeline template (triggerable from dashboard or schedule)
- [x] **CONT-02**: n8n workflow status and recent execution history visible in dashboard
- [x] **CONT-03**: Agent can trigger n8n workflows as part of autonomous task execution (via existing trigger_workflow tool)
- [x] **CONT-04**: Content pipeline outputs (scripts, assets, publish status) tracked in work journal

## Multi-Project Awareness (PROJ)

Agent understands all projects, infrastructure, and their relationships.

- [x] **PROJ-01**: Agent can register and switch between multiple workspace repos with per-project configuration
- [x] **PROJ-02**: Agent maintains per-project context — architecture, conventions, recent changes, key files
- [x] **PROJ-03**: VPS infrastructure state (running containers, services, resource utilization) queryable by agent
- [x] **PROJ-04**: Cross-project operations — agent can reason about how changes in one project affect another

## Financial Tracking (FIN)

Automated cost visibility with budget enforcement and optimization.

- [x] **FIN-01**: LLM API costs tracked per-request with provider, model, token count, and dollar cost attribution
- [x] **FIN-02**: Costs aggregated per goal, per day, and per agent type for budget visibility
- [x] **FIN-03**: Budget alerts triggered when daily or weekly spend exceeds configurable thresholds
- [x] **FIN-04**: Cost optimization suggestions based on usage patterns (e.g., shift routine tasks to cheaper models)

## Dashboard Command Center (DASH)

Dashboard becomes the single pane of glass for the autonomous cofounder.

- [x] **DASH-01**: Work journal page displaying all agent activity in chronological timeline view
- [x] **DASH-02**: Approval queue page showing pending yellow/red tier requests with context and approve/deny actions
- [x] **DASH-03**: Cost dashboard page with charts — daily spend, per-model breakdown, cumulative trend, budget gauge
- [x] **DASH-04**: Multi-project switcher in dashboard header for switching active workspace context
- [x] **DASH-05**: Notification center aggregating agent updates, approval requests, budget alerts, and CI events
- [x] **DASH-06**: Settings page for configuring autonomy tiers per-tool, budget thresholds, and project registrations

## Autonomous Scheduling (SCHED)

The "works while you sleep" loop — agent autonomously picks up and executes planned work.

- [x] **SCHED-01**: Agent picks up planned tasks from backlog and executes them on a recurring BullMQ schedule
- [x] **SCHED-02**: Only one autonomous session runs at a time (distributed lock via Redis)
- [x] **SCHED-03**: Per-session token budget enforced as hard limit — abort mid-execution if exceeded, not just warn
- [x] **SCHED-04**: Self-healing on persistent CI failures — detect pattern, fix code, create PR (yellow-tier approval, 2-cycle confirmation before acting)

## Session Continuity (SESS)

Agent remembers everything across sessions — conversations, decisions, context.

- [x] **SESS-01**: New conversations start with RAG-retrieved context from recent sessions (what was discussed, what was decided, what's in progress)
- [x] **SESS-02**: Agent proactively references past decisions when relevant ("Last time we decided X because Y")
