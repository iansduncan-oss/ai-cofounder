## Phase 8: Memory & Session Foundation

**Goal:** Make the agent remember everything — conversations, decisions, project context — and bring it to every new interaction automatically.

**Requirements:** MEM-01, MEM-02, MEM-03, MEM-04, MEM-05, SESS-01, SESS-02 (7)

**What exists:** RAG pipeline (packages/rag) with pgvector, chunker, embeddings, ingestion queue, retriever with reranking. Orchestrator has auto-retrieval. Memory save/recall tools exist.

**What's new:**
- Automatic conversation ingestion after completion (not manual save_memory)
- Structured decision tagging (what/why/alternatives) as a distinct memory type
- Repo documentation ingestion on project registration
- Pre-task auto-context loading (query RAG before execution starts)
- Background consolidation job that merges related memories
- Session bootstrap — new conversations seeded with recent session context
- Proactive decision referencing in agent responses

**Dependencies:** None (builds on existing RAG infrastructure)

**Success criteria:**
- New conversation receives relevant context from last 3 sessions within first response
- Agent references a past decision unprompted when discussing the same topic
- Conversation ingestion completes within 30s of conversation end

## Phase 9: Autonomy & Approval System

**Goal:** Define what the agent can do freely, what needs approval, and what's forbidden — with a real-time approval flow.

**Requirements:** AUTO-01, AUTO-02, AUTO-03, AUTO-04, AUTO-05 (5)

**What exists:** Approval system (request_approval tool, approvals table, Slack notifications). BullMQ notification queue.

**What's new:**
- Formal three-tier classification (green/yellow/red) with per-tool assignment
- DB table for tier configuration (tool_name, tier, updated_at)
- Pre-execution tier check in orchestrator tool loop
- Approval request delivery with rich context (what tool, what args, why, risk level)
- One-tap approve/deny via Slack interactive message + Discord button
- Dashboard settings page for configuring tiers per-tool
- Approval timeout — auto-deny after configurable period

**Dependencies:** None

**Success criteria:**
- Green tools execute immediately with no delay
- Yellow tools block until approved, with approval delivered to Slack within 2s
- Red tools are refused with explanation, no execution path exists
- Tier config changes from dashboard take effect immediately

## Phase 10: Autonomous Execution Engine

**Goal:** Agent picks up tasks from the backlog and completes them end-to-end — code, test, commit, PR — without human intervention.

**Requirements:** TERM-01, TERM-02, TERM-03, TERM-04, TERM-05 (5)

**What exists:** Orchestrator with 20+ tools, TaskDispatcher, agentic tool loop, git tools, sandbox, workspace tools, PR creation.

**What's new:**
- Autonomous task pickup — agent queries pending tasks by priority and executes next eligible
- Execution orchestration — sequenced tool calls for a full coding workflow (read → edit → test → commit → PR)
- Conventional commit messages with goal/task ID linkage
- Auto-generated PR descriptions from task context + diff summary
- Structured work log entries persisted to DB (new table: `workLogEntries`)
- Execution streamed to dashboard via existing SSE infrastructure

**Dependencies:** Phase 9 (tier checks gate tool execution)

**Success criteria:**
- Agent completes a coding task (edit file, run tests, commit, open PR) with zero human interaction
- Work log entry created for every autonomous execution with accurate duration and cost
- PR description accurately summarizes changes made

## Phase 11: Autonomous Scheduling

**Goal:** Agent works while you sleep — recurring execution loop with safety guardrails.

**Requirements:** SCHED-01, SCHED-02, SCHED-03, SCHED-04 (4)

**What exists:** BullMQ with recurring job scheduler, cron schedule system, monitoring workers.

**What's new:**
- Recurring BullMQ job that triggers autonomous task execution on schedule (configurable interval)
- Redis distributed lock ensuring only one autonomous session at a time
- Per-session token budget as a hard limit — execution aborts cleanly if budget exceeded mid-task
- CI failure detection loop — if same CI check fails 2+ cycles, agent investigates, generates fix, opens PR (yellow-tier)
- Session lifecycle: acquire lock → load context → pick task → execute → log → release lock
- Dashboard toggle to enable/disable autonomous scheduling

**Dependencies:** Phase 10 (needs execution engine), Phase 9 (tier checks for self-healing PRs)

**Success criteria:**
- Autonomous session runs on schedule, picks up a task, completes it, and logs the work
- Second session attempt while one is running is rejected (no dual execution)
- Session aborts cleanly when token budget is exhausted (no partial commits)
- CI self-heal creates a PR fixing a real test failure after 2-cycle confirmation

## Phase 12: Work Journal & Standup

**Goal:** Browsable activity log with daily standup summaries — the "what did the agent do" record.

**Requirements:** JRNL-01, JRNL-02, JRNL-03, JRNL-04 (4)

**What exists:** Tool execution tracking, goal/task tables, briefing system.

**What's new:**
- Work journal DB table aggregating execution records, goal completions, PRs, deployments
- Daily standup generation — LLM summarizes yesterday's journal into standup format
- Standup delivered via existing briefing channel (Slack/Discord/dashboard)
- Journal entries linked to goals/tasks with PR URLs and commit SHAs
- Dashboard work journal page with timeline view, date picker, and search
- Journal auto-populated from work log entries (Phase 10) and monitoring events

**Dependencies:** Phase 10 (work log entries feed the journal)

**Success criteria:**
- Dashboard shows chronological timeline of all agent activity for any date range
- Daily standup is accurate, concise, and delivered before user's configured morning hour
- Journal entries link to the correct PRs and commits

## Phase 13: Financial Tracking

**Goal:** Know exactly what the agent costs — per request, per goal, per day — with budget enforcement.

**Requirements:** FIN-01, FIN-02, FIN-03, FIN-04 (4)

**What exists:** DAILY_TOKEN_LIMIT guardrail, provider health with token tracking, Prometheus metrics for LLM requests.

**What's new:**
- Per-request cost calculation (provider × model × tokens → dollar amount) persisted to DB
- Cost attribution — every LLM call tagged with goal_id, task_id, agent_type
- Aggregation queries: cost by goal, by day, by agent, by model
- Budget threshold configuration (daily limit, weekly limit) stored in DB
- Alert generation when thresholds are crossed (via notification queue)
- LLM-generated cost optimization suggestions based on usage patterns
- API endpoints for cost data (daily summary, goal breakdown, trend)

**Dependencies:** Phase 10 (execution generates cost data with goal attribution)

**Success criteria:**
- Every LLM call has an accurate dollar cost persisted within the request lifecycle
- Dashboard shows cost breakdown by any dimension (day, goal, model, agent)
- Budget alert fires within 1 minute of threshold breach

## Phase 14: Multi-Project Awareness

**Goal:** Agent understands all your projects, infrastructure, and how they relate.

**Requirements:** PROJ-01, PROJ-02, PROJ-03, PROJ-04 (4)

**What exists:** Single workspace (WORKSPACE_DIR), VPS monitoring (SSH health checks), git tools.

**What's new:**
- Project registry DB table (name, repo_url, workspace_path, description, active)
- Project registration flow — clone repo, ingest docs into RAG (MEM-03), store config
- Workspace context switching — orchestrator operates on selected project's workspace
- Per-project RAG namespace — memories/context scoped to project
- VPS infrastructure queries — list containers, service status, resource usage via SSH
- Cross-project reasoning — agent considers dependencies when making changes

**Dependencies:** Phase 8 (RAG for per-project context)

**Success criteria:**
- Agent can switch between 2+ registered projects and maintain correct context for each
- VPS container status is queryable and accurate
- Agent flags potential cross-project impact when making changes

## Phase 15: Content Automations

**Goal:** YouTube pipeline and n8n workflows are managed tasks the agent can trigger, monitor, and report on.

**Requirements:** CONT-01, CONT-02, CONT-03, CONT-04 (4)

**What exists:** n8n integration (trigger_workflow, list_workflows tools), YouTube Shorts pipeline in n8n, pipeline executor.

**What's new:**
- YouTube Shorts pipeline registered as a managed pipeline template with default stages
- n8n execution history API integration — fetch recent runs, status, outputs
- Dashboard n8n panel showing workflow status and execution history
- Content output tracking in work journal (scripts generated, videos published, etc.)
- Scheduled content triggers via autonomous scheduling (Phase 11)

**Dependencies:** Phase 10 (agent triggering), Phase 12 (journal tracking)

**Success criteria:**
- YouTube pipeline is triggerable from dashboard as a one-click managed pipeline
- n8n execution history visible in dashboard with status and timing
- Content outputs appear in work journal with links to published content

## Phase 16: Dashboard Command Center

**Goal:** Dashboard becomes the single pane of glass — work journal, approvals, costs, projects, notifications, and settings all in one place.

**Requirements:** DASH-01, DASH-02, DASH-03, DASH-04, DASH-05, DASH-06 (6)

**What exists:** Dashboard with chat, goals, pipelines, workspace, personas, HUD, briefings.

**What's new:**
- Work journal page (timeline view, date picker, search, goal grouping)
- Approval queue page (pending requests, context preview, approve/deny buttons)
- Cost dashboard page (daily chart, model breakdown pie chart, cumulative trend, budget gauge)
- Multi-project switcher (dropdown in header, context switch on selection)
- Notification center (bell icon → dropdown with agent updates, alerts, approval requests)
- Settings page (autonomy tier config per-tool, budget thresholds, project management, scheduling toggle)

**Dependencies:** Phase 9 (approvals), Phase 12 (journal), Phase 13 (costs), Phase 14 (projects)

**Success criteria:**
- All 6 new dashboard pages/components render correctly with real data
- Approval actions from dashboard update agent execution within 2s
- Settings changes take effect immediately without server restart
