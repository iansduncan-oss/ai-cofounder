    - Constructor: `db: Db`, `llmRegistry: LlmRegistry`, `embeddingService?: EmbeddingService`
    - Method `consolidate(): Promise<{ consolidated: number; created: number }>`:
      1. Fetch all non-consolidated memories via direct DB query (not recallMemories, which requires userId): `db.select().from(memories).where(sql\`metadata->>'consolidated' IS NULL OR metadata->>'consolidated' != 'true'\`).orderBy(desc(memories.createdAt)).limit(200)`. Import `memories` table from schema.
      2. If fewer than 5 memories, log and return { consolidated: 0, created: 0 }.
      3. Group memories by category (decisions, facts, preferences, etc.).
      4. For each category with 3+ memories, use LLM to identify clusters:
         - Build a prompt listing all memories in that category (truncated to 200 chars each).
         - Ask LLM ("simple" task) to group them into clusters of related items and produce a consolidated summary for each cluster.
         - Expected JSON output: `{ clusters: [{ title: string, summary: string, memberIds: string[] }] }`.
      5. For each cluster with 2+ members:
         - Save a new composite memory via `saveMemory(db, { userId: cluster members' userId, category, key: cluster.title, content: cluster.summary, importance: 9, metadata: { consolidated_from: memberIds, consolidated_at: new Date().toISOString() }, embedding })`.
         - Mark each constituent memory with `metadata.consolidated = true` and `metadata.consolidatedInto = newMemoryId` by updating via direct DB update: `db.update(memories).set({ metadata: sql\`metadata || '{"consolidated": true, "consolidatedInto": "${compositeId}"}'\` }).where(eq(memories.id, memberId))`.
      6. Return { consolidated: totalMarked, created: totalNewComposites }.

    IMPORTANT: Use the memories table's existing `metadata jsonb` field for the consolidated flag. No schema migration needed.
